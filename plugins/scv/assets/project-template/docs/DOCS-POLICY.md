# 문서 관리 정책

이 문서는 `.scv/docs` 아래 문서의 책임, 갱신 조건과 검토 규칙을 정의한다.
한 사실에는 하나의 권위 원본만 두고, 다른 문서에서는 링크와 필요한 맥락만
제공한다.

## 문서별 책임

| 경로 | 책임 | 주요 갱신 시점 |
|---|---|---|
| `SYSTEM.md` | 시스템 경계, 구성요소와 데이터 흐름 | 구조 또는 책임 변경 |
| `CHANGES.md` | 사용자에게 의미 있는 변경 | 사용자 동작 또는 호환성 변경 |
| `work-items/` | 승인된 요청·계획과 완료 핸드오프의 snapshot | 계획 승인과 완료 gate |
| `adr/` | 장기적으로 유지할 아키텍처 결정 | 중요한 대안 선택과 정책 변경 |
| `investigations/` | 결정에 사용한 조사, 실험과 근거 | 비정상 동작 조사와 기술 검증 |
| `quality/VERIFICATION.md` | 검증 계층과 표준 실행 방법 | 테스트 체계 또는 명령 변경 |
| `quality/KNOWN-GAPS.md` | 알려진 품질 공백과 개선 계획 | 공백 발견, 완화 또는 해소 |

## 제어 데이터와 문서의 분리

- `.scv/tasks`는 재개 가능한 실행 상태의 원본이며 사람이 직접 편집하지 않는다.
- `.scv/docs/work-items`는 승인 또는 완료 gate에서 발행한 읽기용 snapshot이다.
- 모델 원시 출력, 전체 대화 기록, cache와 임시 파일은 장기 문서로 복사하지 않는다.
- 실행 증거는 태스크 저장소에 두고 문서에는 결과 요약과 필요한 참조만 남긴다.

## 작성 규칙

- 저장소와 실행 결과에서 확인한 사실, 합의한 결정, 아직 확인되지 않은 가정을
  구분한다.
- 자격증명, 비밀값, 환경변수 원문, 개인 식별 정보와 장비별 절대 경로를 기록하지
  않는다.
- 사용자에게 의미 있는 변경은 구현과 같은 작업에서 문서화한다.
- 동일한 내용을 여러 파일에 복제하지 않고 권위 문서로 연결한다.
- 조사 결과가 장기 결정으로 이어지면 조사 문서와 ADR을 서로 연결한다.

## 검토 규칙

- 계획 승인 전에는 영향을 받는 문서와 필요한 ADR을 식별한다.
- 검증 단계에서는 코드, 설정, 테스트와 문서가 같은 동작을 설명하는지 확인한다.
- 승인된 ADR은 제자리에서 의미를 바꾸지 않는다. 변경된 결정은 새 ADR을 추가하고
  `supersedes` 관계를 기록한다.
- 문서 변경도 다른 산출물과 동일하게 승인 범위와 검증 근거를 가져야 한다.
